# agentguard architecture

## Goals

1. **Make every agent tool call an explicit authorization decision** — no implicit "trust the agent".
2. **Make every decision auditable** — structured JSONL log of who, what, when, why.
3. **Make agent-to-agent delegation safe** — scoped, time-boxed, signed tokens.
4. **Make policies first-class artifacts** — Cedar, with schema validation, written by humans or generated by LLMs.
5. **Stay close to Cedar** — agent-specific abstractions on top, not a replacement.

## The server

`agentguard-server` is the AuthZEN-compatible HTTP + gRPC PDP. It exposes:

- `POST /access/v1/evaluation` — single decision (AuthZEN draft).
- `POST /access/v1/evaluations` — batch (cap 100 per call).
- `GET /healthz` / `/readyz` — Kubernetes probes.
- `GET /metrics` — Prometheus text snapshot.
- `agentguard.v1.AccessEvaluation` (gRPC) — same surface over HTTP/2.

All `/access/v1/*` endpoints are protected by `auth_layer`:

- `AuthConfig::Disabled` — no auth, refused on a public listener unless
  `AGENTGUARD_ALLOW_LOOPBACK_BYPASS=1` is set.
- `AuthConfig::ApiKey` — `Authorization: Bearer <raw>` against a JSON
  `ApiKeyStore`. Argon2id is the verification path.

A `PolicyWatcher` polls `store_root` every 500 ms; on each event the
decision cache is invalidated and `policy_reload_total` is bumped. The
server also installs a SIGHUP handler (Unix) so operators can force a
refresh without touching the filesystem.

## The request model

Every tool call is normalized into a Cedar `(Principal, Action, Resource, Context)` tuple:

- **Principal** is either a `User::"<id>"` (human) or `Agent::"<id>"` (AI agent). Agents can have a `parent` link for sub-agent chains.
- **Action** is `Action::"ToolCall::<tool_name>"` (optionally `::"<operation>"`). For example, `Action::"ToolCall::s3::PutObject"`.
- **Resource** is whatever the tool targets — a `Mailbox`, a `Repository`, a `Document`. You define your domain types in the schema.
- **Context** carries tool args (flattened to top level for schema validation) plus a `session` record with `ip`, `user_agent`, `mfa`, `ts`.

## The decision log

Every authorization call writes a `DecisionRecord` to `.audit/decisions.jsonl`:

```json
{
  "id": "uuid",
  "timestamp": "2026-07-14T12:34:56Z",
  "effect": "allow" | "deny",
  "policies": ["policy_id", ...],
  "principal": "alice",
  "action": "send_email",
  "resource": "alice@acme",
  "reasons": ["..."],
  "request_id": "...",
  "session_id": "..."
}
```

This is the audit trail. It's append-only, JSONL, and easy to ship to your log
aggregator (Datadog, Splunk, Loki, etc).

## Delegation

When agent A calls agent B, B shouldn't inherit A's full identity. agentguard
mints **scoped delegation tokens**:

```json
{
  "iss": "Agent::\"research\"",
  "sub": "Agent::\"summarizer\"",
  "aud": "agentguard",
  "iat": 1700000000,
  "exp": 1700000900,
  "allowed_actions": ["ToolCall::send_email"],
  "resource_patterns": ["Mailbox::*"]
}
```

Tokens are Ed25519-signed. Compact form: `base64url(payload).base64url(sig).key_id`.

The verifier checks:
1. Signature validity
2. `exp` not in the past
3. `nbf` not in the future (optional)
4. The action/resource being requested is in the allowed scope

## NL policy generation

`agentguard gen "<natural language>"` calls an LLM with:

1. Your schema (`schema.cedarschema`) as context
2. A constrained system prompt instructing Cedar-only output
3. Your natural-language requirement

The output is validated against your schema before being written to disk. If
the LLM produces invalid Cedar or policies that don't type-check, the command
fails and you refine the prompt.

This forms a tight feedback loop — Cedar's validator is the ground truth.

## Schema as source of truth

The Cedar schema does two important jobs:

1. **Validates policies at authoring time.** If a policy references an entity
   type that doesn't exist, or uses a wrong attribute type, validation fails
   immediately — not at runtime.
2. **Validates context at request time.** The action's declared context shape
   is enforced when the SDK builds a request. If `send_email` requires
   `subject: String`, sending a request without it fails fast.

## Local-first

All state lives on disk:

- `.agentguard/schema.cedarschema` — schema
- `.agentguard/policies/*.cedar` — policies (one per file)
- `.audit/decisions.jsonl` — decision log
- (delegation tokens are passed at runtime, not stored)

This makes it trivial to:

- Version-control your policies (`git diff .agentguard/policies/`)
- Test policies locally (`agentguard sim req.json`)
- Review changes in PRs
- Migrate from local to hosted by syncing the same files

The future hosted service (out of scope for v1) would sync these directories
across a team.
