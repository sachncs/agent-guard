# agentguard architecture

## Goals

1. **Make every agent tool call an explicit authorization decision** — no implicit "trust the agent".
2. **Make every decision auditable** — structured JSONL log of who, what, when, why.
3. **Make agent-to-agent delegation safe** — scoped, time-boxed, signed tokens.
4. **Make policies first-class artifacts** — Cedar, with schema validation, written by humans or generated by LLMs.
5. **Stay close to Cedar** — agent-specific abstractions on top, not a replacement.
6. **Operate in production** — observability, retry, drain, health checks, runbook.

## The server

`agentguard-server` is the AuthZEN-compatible HTTP + gRPC PDP. It exposes:

- `POST /access/v1/evaluation` — single decision (AuthZEN draft).
- `POST /access/v1/evaluations` — batch (cap 100 per call).
- `GET /healthz` / `/readyz` — Kubernetes probes (healthz always 200;
  readyz requires policies loaded + audit log open).
- `GET /metrics` — Prometheus text snapshot.
- gRPC: `agentguard.v1.AccessEvaluation` (enable with
  `AGENTGUARD_GRPC_LISTEN`).

All `/access/v1/*` endpoints are protected by `auth_layer`:

- `AuthConfig::Disabled` — no auth, refused on a non-loopback listener
  unless `AGENTGUARD_ALLOW_LOOPBACK_BYPASS=1` is set.
- `AuthConfig::ApiKey { path }` — `Authorization: Bearer <raw>` against a
  JSON `ApiKeyStore`. Argon2id is the verification path.

A `PolicyWatcher` polls `store_root` every 500 ms; on each event the
decision cache is invalidated and `policy_reload_total` is bumped. The
server also installs a SIGHUP handler (Unix) so operators can force a
refresh without touching the filesystem.

Graceful shutdown is bounded at 30 s (axum's default waits forever for
in-flight requests). After SIGTERM/SIGINT the server stops accepting new
connections, drains in-flight requests for at most 30 s, then exits.

## The request model

Every tool call is normalized into a Cedar `(Principal, Action, Resource, Context)` tuple:

- **Principal** is either a `User::"<id>"` (human) or `Agent::"<id>"` (AI
  agent). Agents can have a `parent` link for sub-agent chains. The
  AuthZEN `subject.entity_type` is honored — `"User"` produces a
  `User` principal, `"Agent"` produces an `Agent` principal, anything
  else returns 400.
- **Action** is `Action::"ToolCall::<tool_name>"` (optionally
  `::"<operation>"`). For example, `Action::"ToolCall::s3::PutObject"`.
- **Resource** is whatever the tool targets — a `Mailbox`, a `Repository`,
  a `Document`. You define your domain types in the schema.
- **Context** carries tool args (flattened to top level for schema
  validation) plus a `session` record with `ip`, `user_agent`, `mfa`,
  `ts`.

## The decision log

Every authorization call writes a `DecisionRecord` to `.audit/decisions.jsonl`:

```json
{
  "id": "uuid",
  "timestamp": "2026-07-14T12:34:56Z",
  "effect": "allow" | "deny",
  "policies": ["policy_id", ...],
  "principal": "alice",
  "action": "send_email",
  "resource": "alice@acme",
  "reasons": ["..."],
  "request_id": "...",
  "session_id": "..."
}
```

The log is **append-only** with one of two modes:

- **Plain JSONL** (no secret): fast, debuggable, but tamper-evident
  only against honest mistakes.
- **Hash-chained** (`AGENTGUARD_CHAIN_SECRET=<file>`): each record
  carries `prev_hash` + `record_hash` = `HMAC-SHA256(root_key, payload)`;
  `verify_chain` confirms no record was inserted, removed, or
  reordered. Verified offline with `agentguard audit verify`.

On startup the chain head is rehydrated from disk; corruption is
reported via `tracing::warn!` rather than silently resetting. The
chain head advance and the file `write_all + sync_all` happen under a
single lock, so a crash leaves the on-disk file at the exact head
recorded (no off-by-one).

`fsync` is called on every append in both modes — a power loss
between flush and the kernel page-cache flush cannot lose records.

## Delegation

When agent A calls agent B, B shouldn't inherit A's full identity.
`agentguard` mints **scoped delegation tokens** as standard JWS
(RFC 7515) using Ed25519:

```json
{
  "iss": "Agent::\"research\"",
  "sub": "Agent::\"summarizer\"",
  "aud": "agentguard",
  "iat": 1700000000,
  "exp": 1700000900,
  "allowed_actions": ["ToolCall::send_email"],
  "resource_patterns": ["Mailbox::*"]
}
```

Compact form: `base64url(header).base64url(payload).base64url(sig)`.

The verifier (`DelegationVerifier::verify`) checks, in order:

1. JWS parses cleanly (3 segments, valid base64url, valid JSON).
2. `alg` is `EdDSA` and `kid` is registered. HS256 + Ed255SA
   combinations (algorithm-confusion attacks) are rejected
   explicitly.
3. Signature is valid against the registered key.
4. `exp` is in the future (with a configurable clock skew,
   default 60 s).
5. `nbf` is in the past (with the same clock skew).
6. `aud` matches the configured audience.

Optional structured `constraints` (Cedar-style path expressions
evaluated against the request `context`) further narrow the token's
applicability.

## NL policy generation

`agentguard gen "<natural language>"` calls an LLM with:

1. Your schema (`schema.cedarschema`) as context
2. A constrained system prompt instructing Cedar-only output
3. Your natural-language requirement

The output is validated against your schema before being written to
disk. If the LLM produces invalid Cedar or policies that don't
type-check, the command fails and you refine the prompt.

The HTTP client has a 60 s request / 10 s connect timeout and retries
up to 3 times on 5xx / connect / timeout failures with exponential
backoff (250 ms / 500 ms / 1 s cap).

This forms a tight feedback loop — Cedar's validator is the ground truth.

## Schema as source of truth

The Cedar schema does two important jobs:

1. **Validates policies at authoring time.** If a policy references an entity
   type that doesn't exist, or uses a wrong attribute type, validation fails
   immediately — not at runtime.
2. **Validates context at request time.** The action's declared context shape
   is enforced when the SDK builds a request. If `send_email` requires
   `subject: String`, sending a request without it fails fast.

## Local-first

All state lives on disk:

- `.agentguard/schema.cedarschema` — schema
- `.agentguard/policies/*.cedar` — policies (one per file)
- `.audit/decisions.jsonl` — decision log (+ `.chainid` sidecar)
- (delegation tokens are passed at runtime, not stored)

This makes it trivial to:

- Version-control your policies (`git diff .agentguard/policies/`)
- Test policies locally (`agentguard sim req.json`)
- Review changes in PRs
- Migrate from local to hosted by syncing the same files

## Configuration

All settings have sensible defaults; overrides come from CLI flags
or environment variables. The full table is in
`docs/operations/runbook.md`. Highlights:

- `AGENTGUARD_LISTEN` — listen address (default `tcp://127.0.0.1:8443`)
- `AGENTGUARD_STORE` — policy directory (default `.agentguard`)
- `AGENTGUARD_AUDIT` — audit log path (default `.audit/decisions.jsonl`)
- `AGENTGUARD_CHAIN_SECRET` / `--secret-file` — HMAC chain secret
  (hex / base64 / raw bytes; auto-detected)
- `AGENTGUARD_AUTH` — `disabled` or `apikey:<path>`
- `AGENTGUARD_AUTH_KEY_FILE` — companion to `--auth apikey`
- `AGENTGUARD_GRPC_LISTEN` — optional gRPC listen address
- `AGENTGUARD_ALLOW_LOOPBACK_BYPASS` — escape hatch for embedders
- `AGENTGUARD_CACHE_TTL` — decision cache TTL (humantime, default 60 s)
- `AGENTGUARD_CACHE_CAPACITY` — decision cache size (default 10 000)
- `AGENTGUARD_JWKS_REFRESH` — JWKS refresh interval (humantime, default 30 s)
